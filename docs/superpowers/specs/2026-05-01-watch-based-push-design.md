# 关注项监控推送改造设计

日期：2026-05-01

## 1. 目标

将当前以“热点新闻 / RSS 聚合推送”为核心的项目，改造成以“关注项监控推送”为核心的项目。

新的系统不再默认推送新闻和 GitHub 榜单，而是围绕用户主动定义的关注项运行。用户可以输入一个问题，或者一个名称，例如：

- `iPhone 16 Pro 什么时候降价`
- `范冰冰微博有没有更新`
- `某品牌活动页有没有变化`

系统需要支持以下能力：

- 用户可直接提供一个或多个网址作为监控来源
- 用户也可以只提供问题 / 名称，由系统自动搜索候选来源
- 自动找到的来源默认进入待确认状态，确认后再正式监控
- 每次定时运行时，逐个关注项抓取、解析、比对、生成结果
- 推送内容以“关注项结果”为核心，而不是新闻榜单

## 2. 非目标

第一版不做以下内容：

- 不继续维护“新闻 / RSS / GitHub 榜单”为默认主路径
- 不追求支持所有网站的零配置精确解析
- 不实现复杂网页登录态管理
- 不做可视化后台页面
- 不做全自动高风险来源确认，默认采用半自动确认

## 3. 现状与约束

当前项目的稳定主链路已经存在：

- `trendradar/__main__.py` 负责主流程、调度决策、推送触发
- `trendradar/context.py` 负责配置、存储、通知、调度封装
- `trendradar/core/scheduler.py` 已经具备成熟的时间段调度能力
- `trendradar/notification/*` 已经具备成熟的多渠道推送能力
- `trendradar/storage/*` 已经具备本地 / 远程持久化能力

因此本次改造不应重写调度和通知，而应新增一条“关注项采集与判断”主链，并逐步替换现有新闻主链的默认入口。

## 4. 方案选型

### 方案 A：纯规则型监控

每个关注项都要求配置精确规则，例如价格选择器、最新一条动态选择器。

优点：

- 结果最稳定
- 误报最少

缺点：

- 配置成本高
- 每种网站都要单独适配
- 与“输入一个问题或名称即可”的目标不一致

### 方案 B：纯通用页面对比型

所有关注项统一抓网页文本摘要，对比前后差异决定是否推送。

优点：

- 通用性高
- 接入快

缺点：

- 商品页价格提取不稳定
- 动态页容易被广告、推荐、时间戳干扰
- 误报率偏高

### 方案 C：混合型

统一抽象“关注项”，但解析方式分类型：

- `price`：价格监控
- `feed_update`：主页 / 动态 / 最新内容更新监控
- `generic_page_change`：通用页面变化监控

推荐采用此方案。

原因：

- 保留“问题 / 名称”作为用户入口
- 对商品降价和主页更新这两类高频需求提供更稳定的专用处理
- 保留通用网页监控兜底能力
- 与现有调度和推送体系兼容，改造成本可控

## 5. 核心设计

### 5.1 关注项模型

第一版引入统一的 `watch` 概念。每个关注项建议包含以下字段：

- `id`：关注项唯一标识
- `title`：用户输入的问题或名称
- `description`：可选补充说明
- `mode`：`manual` 或 `auto`
- `source_urls`：用户手动提供的网址列表
- `resolved_sources`：系统识别并确认后的实际监控来源
- `watch_type`：`price` / `feed_update` / `generic_page_change`
- `push_policy`：`silent` / `report_no_change`
- `status`：`pending_confirm` / `active` / `paused` / `error`
- `last_snapshot`：最近一次有效抓取结果
- `last_check_at`：最近一次检查时间
- `last_change_at`：最近一次确认发生变化的时间
- `last_push_at`：最近一次推送时间
- `error_message`：最近一次错误信息

### 5.2 来源发现

来源发现分三层：

1. 手动直连  
用户提供网址时，直接以该网址作为候选来源。

2. 自动搜索候选  
用户只提供问题 / 名称时，系统搜索多个候选页面。

3. 半自动确认  
系统将候选来源记录为待确认项，用户确认后才转为正式监控。

第一版默认采用半自动确认，避免自动搜索误命中导致错误监控。

### 5.3 关注项类型判定

第一版只支持三种类型：

#### `price`

适用于商品详情页、活动页、订阅页等包含明确价格字段的页面。系统提取当前价格，与上一次价格比较。

#### `feed_update`

适用于微博主页、博客主页、作者主页、频道页、公告页等“最新一条内容驱动”的页面。系统提取最新内容的唯一标识并比对。

#### `generic_page_change`

适用于暂时无法精确归类，但仍希望监控的网页。系统提取主体内容摘要或指纹，判断是否发生实质变化。

### 5.4 快照与变化判断

#### `price`

快照内容至少包含：

- 页面标题
- 商品名
- 当前价格
- 可选原价 / 券后价 / 促销标签
- 抓取时间

判定规则：

- 当前价格 < 上次价格：判定为降价
- 当前价格 == 上次价格：判定为未降价
- 当前价格 > 上次价格：判定为涨价，但第一版默认不单独强调，可作为“未降价”处理

#### `feed_update`

快照内容至少包含：

- 页面标题
- 最新一条内容的唯一标识
- 最新一条内容的标题或摘要
- 最新一条内容的发布时间
- 链接

判定规则：

- 最新内容标识变化：判定为有更新
- 未变化：判定为无更新

#### `generic_page_change`

快照内容至少包含：

- 页面标题
- 主体文本摘要
- 文本指纹
- 抓取时间

判定规则：

- 主体文本变化低于阈值：视为无变化
- 主体文本发生实质变化：视为有变化

### 5.5 推送规则

每个关注项独立配置 `push_policy`：

- `silent`：无变化不推
- `report_no_change`：无变化也推

行为定义：

- 商品价格类可以选择 `report_no_change`，推送“未降价”
- 微博 / 动态类通常建议 `silent`，只有更新才推
- 通用页面变化类默认建议 `silent`

推送文案示例：

- `京东 iPhone 16 Pro 已降价：7999 -> 7599`
- `京东 iPhone 16 Pro 暂未降价，当前仍为 7999`
- `范冰冰微博有新动态：<标题或摘要>`
- `范冰冰微博暂无更新`

## 6. 模块架构改造

### 6.1 新增模块

建议新增 `trendradar/watch/` 包，拆分如下：

- `models.py`
  定义关注项、来源、快照、结果等数据结构

- `resolver.py`
  负责根据问题 / 名称和可选 URL 产出候选来源

- `fetcher.py`
  负责抓取网页内容，返回统一原始页面对象

- `detectors/price.py`
  价格类解析与比对

- `detectors/feed_update.py`
  动态更新类解析与比对

- `detectors/generic_change.py`
  通用页面变化检测

- `service.py`
  统一调度关注项检查，输出本轮结果

- `formatter.py`
  将关注项检查结果转成通知展示结构

### 6.2 持久化

当前 `trendradar/storage/*` 已适合继续复用，但需要新增关注项专用表。

建议新增两类存储：

- `watches`
  保存关注项定义、状态、来源、推送策略

- `watch_snapshots`
  保存每次抓取后的快照与结果摘要

这样可以保留：

- 最近状态
- 历史变化轨迹
- 后续调试能力

### 6.3 主流程接入

`trendradar/__main__.py` 当前主流程是：

- 调度解析
- 抓取新闻 / RSS
- 生成报告
- 推送通知

第一版改为新增一条关注项主流程：

1. 调度解析
2. 读取所有 `active` 的关注项
3. 逐个执行来源抓取与变化判断
4. 汇总本轮关注项结果
5. 生成关注项推送内容
6. 复用现有 `notification/dispatcher.py` 推送

为降低风险，建议保留旧主链但关闭默认入口，通过配置切换到关注项模式。

### 6.4 通知层复用

`trendradar/notification/*` 目前已经成熟，不建议重写。

第一版只需要把“新闻报告数据结构”旁边新增一套“关注项结果数据结构”，再在渲染层做最小适配：

- 标题从“热点分析报告”改成“关注项监控报告”
- 列表项从“关键词 / 新闻列表”改成“关注项 / 检测结果”
- 保留现有多渠道发送、批量拆分、翻译开关、HTML 能力

## 7. 配置设计

第一版建议新增独立配置文件，例如：

- `config/watches.yaml`

内容结构建议为：

- 全局开关
- 来源发现模式
- 默认推送策略
- 关注项列表

每个关注项包含：

- 名称或问题
- 可选 URL 列表
- 可选类型
- 无变化推送策略
- 是否启用

这样能避免把大量关注项直接塞进原 `config.yaml`。

## 8. 错误处理

系统需要区分“无变化”和“检查失败”。

必须避免以下误判：

- 商品页抓取失败却被误报为“未降价”
- 主页无法访问却被误报为“暂无更新”
- 页面结构变化导致解析失败却继续输出旧结果

建议规则：

- 请求失败：记录错误，不输出“无变化”结论
- 解析失败：关注项标记为 `error` 或本轮 `check_failed`
- 连续失败达到阈值：允许后续扩展为单独告警

## 9. 测试策略

第一版至少覆盖以下测试：

- 手动 URL 的价格关注项可正确比较价格变化
- 手动 URL 的动态关注项可正确识别最新一条更新
- 自动发现来源后可进入待确认状态
- 未确认来源不会进入正式推送
- `silent` 策略在无变化时不推送
- `report_no_change` 策略在无变化时输出明确结果
- 抓取失败不会误报为“无变化”
- 定时调度仍能复用当前 `Scheduler`
- 多渠道通知仍能复用当前 `NotificationDispatcher`

## 10. 第一版边界

第一版明确只保证以下场景：

- 关注项入口支持“问题”或“名称”
- 支持手动 URL
- 支持自动搜索候选来源
- 默认走半自动确认
- 支持 `price`、`feed_update`、`generic_page_change`
- 支持“无变化静默”与“无变化汇报”
- 支持复用现有调度和通知

第一版不保证：

- 所有商品站点都能稳定提取价格
- 所有社交平台都能无适配抓到最新动态
- 需要登录的网站都能直接监控
- 无确认的自动来源一定准确

## 11. 迁移策略

建议分阶段迁移：

### 阶段 1

新增关注项模型、存储、基础配置、主流程骨架，保留旧新闻逻辑。

### 阶段 2

完成 `price` 与 `feed_update` 两类检测器，并打通推送。

### 阶段 3

补充 `generic_page_change`，接入自动搜索候选来源与待确认机制。

### 阶段 4

根据实际使用情况决定是否彻底移除新闻 / RSS 默认路径，或保留为兼容模式。

## 12. 推荐结论

你的想法可以实现，而且适合在现有项目上渐进式落地。

最佳做法不是继续围绕“新闻源”打补丁，而是把系统上升为“关注项监控推送器”：

- 用户输入的是问题或名称
- 系统维护的是关注项和来源
- 定时任务检查的是关注项状态
- 推送出去的是关注项结果

这条路线与当前项目已有的调度、通知、存储能力兼容，能以较低风险完成转型，并且后续还能扩展到价格、动态、公告、专题页面等更多场景。
